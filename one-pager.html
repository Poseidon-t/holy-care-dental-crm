Data Safety
Medical Records Protected & Backed Up
Patient medical records are critical. The system is built with multiple layers of protection to ensure data is never lost, corrupted, or accessed without authorization.
ProtectionHow It Works
Automated Daily Backups Full database snapshot uploaded to a private GitHub repository every 24 hours automatically. Last 30 daily backups retained, plus a latest copy for instant restore.
Persistent Volume Storage Database lives on a dedicated persistent volume on Railway — survives server restarts, deployments, and container recycling.
Crash-Resistant Database SQLite with WAL (Write-Ahead Logging) mode and synchronous = FULL — data is written to disk before confirming. Zero data loss even on unexpected shutdowns.
Atomic Transactions Patient creation and related operations run inside database transactions — either everything saves or nothing does. No partial or corrupted records.
Off-Site Backup Storage Backups stored in a private GitHub repo separate from the server. Even if the server is completely lost, all patient data can be restored from the latest backup.
Security
Built Secure by Default
FeatureWhat It Solves
JWT Auth + scrypt Passwords Patient data behind login. Passwords cryptographically hashed with random salts and verified using timing-safe comparison.
Middleware Route Protection Every dashboard/API request verified server-side. No data leaks without a valid session cookie.
Login Rate Limiting 5 login attempts per 15 minutes per IP address. Prevents brute-force password attacks.
Chat Rate Limiting 10 messages/min per IP. Prevents spam and protects AI API quota.
Input Validation All patient data validated with length limits, type checks, and sanitization before storage. Signatures capped at 2MB.
Registration Link Expiry Remote patient registration links expire after 24 hours and can only be used once. Prevents stale or reused links.
Security Headers HSTS, CSP, X-Frame-Options, and other headers prevent XSS, clickjacking, and content injection attacks.
Env-based Secrets API keys and passwords stored as environment variables — never in source code.
